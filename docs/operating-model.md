# Operating Model

Project Atlas uses a simple path from problem discovery to delivered outcome.

```text
Problem -> Proposal -> Review -> Project -> Delivery -> Outcome
```

## 1. Problem

A problem describes something worth solving. It should be written in plain language and should not assume a specific technical solution too early.

Examples:

- Older people struggle to access digital-only public services.
- Small charities cannot afford modern automation.
- Open-source maintainers are overwhelmed by issue triage.

## 2. Proposal

A proposal describes a possible intervention, tool, workflow, dataset, research project, or open-source build.

## 3. Review

The community improves the proposal by adding research, existing solutions, risks, scope, and feasibility analysis.

## 4. Project

A proposal becomes a project when it has enough clarity, value, and contributor interest.

## 5. Delivery

A project should have maintainers, tasks, documentation, and a visible roadmap.

## 6. Outcome

An outcome is something usable: code, documentation, research, a deployed service, a reusable template, or a public specification.

## Anti-patterns

Project Atlas should avoid:

- Endless discussion with no decision
- Popularity contests with no feasibility review
- Duplicate projects with no consolidation
- Building before understanding the problem
- Over-engineering the platform before the community exists

## Initial tooling

During the founding phase, use:

- GitHub Issues for structured proposals and tasks
- GitHub Discussions for broader debate and polls
- GitHub Projects for roadmap/status tracking
- Markdown documents for governance and process
- Static website for public-facing explanation
