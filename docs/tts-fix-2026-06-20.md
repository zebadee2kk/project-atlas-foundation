# TTS Fix Log — 2026-06-20

## Problem

Voice replies from Hermes stopped working. The failure trace was:

1. Built-in `text_to_speech` tool attempted to use the `piper` provider.
2. The local Piper voice cache at `~/.hermes/cache/piper-voices/en_US-lessac-medium.onnx` was actually a PyTorch model blob misnamed as `.onnx`, causing ONNX runtime protobuf parsing to fail.
3. Earlier troubleshooting also used the wrong CLI entrypoint: `hermes text-to-speech` does not exist. The Hermes CLI surface does not include a TTS subcommand; TTS is invoked through the built-in tool/provider layer, not the `hermes` command tree.

## Root cause

- Corrupted / mismatched Piper voice cache file downloaded as PyTorch format instead of a valid Piper-compatible ONNX model.
- Config had `tts.provider` set to `piper`, so every TTS attempt loaded the broken cache.

## Fix applied

- Removed the bad cache:
  - `~/.hermes/cache/piper-voices/en_US-lessac-medium.onnx`
  - `~/.hermes/cache/piper-voices/en_US-lessac-medium.onnx.json`
- Changed `tts.provider` from `piper` to `edge` in `~/.hermes/config.yaml`, which uses Microsoft Edge neural voices (`en-US-AriaNeural`).
- Backed up the original config before editing.
- Verified with the proper in-session `text_to_speech` tool; it generated an MP3 successfully via the Edge provider.

## Result

Voice replies are functional again. Output was produced as MP3 via Edge TTS. If true Telegram voice bubbles (Opus/OGG) are desired later, that is a routing/format follow-up, not a blocker.

## Lessons

- Piper voice caches can end up in the wrong format; clearing the cache and retrying is safer than attempting to repair the model file.
- Do not invoke TTS via `hermes text-to-speech`; use the built-in tool/provider path instead.
- Provider fallback should be checked before assuming a deep model/install issue.
